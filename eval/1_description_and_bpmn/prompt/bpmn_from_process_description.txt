# TASK DESCRIPTION

You are an expert in Business Process Model and Notation (BPMN 2.0) and Signavio's specific XML structure.

Your task is to transform a structured, text-based **process description** into an executable process model in `.bpmn` format that is valid according to the BPMN 2.0 modeling standard and perfectly compatible with Signavio.

## MAPPING RULES (NARRATIVE TO BPMN)

The input text is a pre-processed, structured narrative with strict conventions. You must accurately translate these textual cues into their corresponding BPMN elements.

**PART 1: SEMANTIC LOGIC**

1. **Pools & Lanes (Actors)**
* *Input cue*: "The process contains the following actors: [X, Y] (with lanes: A, B)"
* *Action*: Create a `<participant>` (Pool) and a corresponding `<process>` for each top-level actor. Map any listed subdivisions to `<laneSet>` and `<lane>` elements within that process.


2. **Start and End Events**
* *Input cue*: "The process starts with [trigger]" / "The process for the [Actor] ends with [Event]."
* *Action*: Add `<startEvent>` and `<endEvent>` nodes in the respective actor's pool.


3. **Tasks (Activities)**
* *Input cue*: "[Actor] shall [activity]..."
* *Action*: Create a `<task>` element assigned to the correct pool/lane. Connect tasks strictly in the order they are described using `<sequenceFlow>`.


4. **Exclusive Gateways (XOR)**
* *Input cue*: "[Actor] shall assess whether [condition]. If [condition is true]... If [condition is false]..." or "[Actor] may [activity] if..."
* *Action*: Insert an `<exclusiveGateway>`. Create outgoing `<sequenceFlow>` elements representing the branching logic (the "true", "false", or "except where" paths). Ensure the sequence flows merge back appropriately if the paths reconverge.


5. **Parallel Gateways (AND)**
* *Input cue*: "[Actor] shall carry out the following activities in parallel..."
* *Action*: Insert a `<parallelGateway>` (diverging), route sequence flows to the parallel tasks, and insert a converging `<parallelGateway>` when the narrative states "All activities shall be completed before...".


6. **Message Flows (Inter-actor Communication)**
* *Input cue*: "[Actor A] shall transmit [info] to [Actor B]" AND "Upon receiving [info], [Actor B]..."
* *Action*: Create `<messageFlow>` elements connecting the boundary of the sending task/event in Actor A's pool to the receiving task/event in Actor B's pool.



**PART 2: DIAGRAM INTERCHANGE (VISUAL LAYOUT) - CRITICAL**
You MUST generate the `<bpmndi:BPMNDiagram>` section. A model without visual coordinates is broken.
7. **Shapes (Nodes)**

* *Action*: For EVERY Pool, Lane, Task, Event, and Gateway created in Part 1, you MUST create a corresponding `<bpmndi:BPMNShape>` with absolute X/Y `<omgdc:Bounds>`.

8. **Edges (Arrows)**
* *Action*: For EVERY `<sequenceFlow>` and EVERY `<messageFlow>` created in Part 1, you ABSOLUTELY MUST create a corresponding `<bpmndi:BPMNEdge>` linking to the flow's ID via the `bpmnElement` attribute.
* Each `<bpmndi:BPMNEdge>` MUST contain at least two `<omgdi:waypoint>` elements (X/Y coordinates connecting the source shape to the target shape). **Do not omit sequence flow edges under any circumstances.**



## Extraction procedure

Let’s carry out the extraction in three steps.

1. Extract all mentions of activities and actors.
2. Reading the input text again extract all relations between extracted mentions.
3. Model the extracted information as a process in BPMN 2.0 Notation. **You must include the complete layout information (DI part), explicitly ensuring no `<sequenceFlow>` is missing its `<bpmndi:BPMNEdge>` counterpart.**

# RESTRICTIONS

## Additional considerations

* If there is no action, there cannot be any actor in the sentence.
* Use the sentences’ context, but relations must not cross sentence boundaries.

## Disambiguation hints

* In the following there is only one activity (print claim), the rest are details:
“The claims officer prints the claim by clicking [PRINT].”

## Format instructions

The output should be an executable .bpmn format (XML). Ensure all XML tags are properly closed and valid.

## Other restrictions
- **Gateway attributes**: Every `<exclusiveGateway>` must include `isMarkerVisible="true"`.
  Every gateway (both exclusive and parallel) must include `gatewayDirection="Diverging"`
  or `gatewayDirection="Converging"` as appropriate.
- **Sequence flow immediacy**: Every `<sequenceFlow>` must include `isImmediate="true"`.
- **Branch condition labels**: On XOR gateway outgoing flows, set the `name` attribute to
  the branch condition (e.g., `name="yes"` / `name="no"`).
- **Intermediate message catch events**: When the process description states "[Actor] shall
  receive [X]" mid-process (not as the first event), use `<intermediateCatchEvent>` with a
  `<messageEventDefinition>` child, not a task or start event.
- **Message flow targeting**: When a message is sent to an actor with no specific receiving
  element (i.e., the receiving actor has no modeled intermediate event), the `targetRef` of
  `<messageFlow>` must point to the `<participant>` element of that pool, not a task.
- **Label positioning**: Include `<signavio:signavioLabel>` with appropriate `align`, `ref`,
  and coordinate attributes on all gateways, intermediate events, and end events to ensure
  correct label rendering in Signavio.
